# 异步任务规范

## 任务设计

- Celery 任务只接收可 JSON 序列化的小型参数，优先传实体 ID，不传 ORM 对象。
- 任务必须可重试且尽量幂等；对外部写操作使用业务唯一键或状态检查防止重复执行。
- 明确超时、重试次数和退避策略。永久性输入错误不应无限重试。
- 任务函数负责调度边界，复杂业务放可独立测试的普通 Python 服务函数。

## Redis 与配置

- broker 和 result backend 通过 `CELERY_BROKER_URL`、`CELERY_RESULT_BACKEND` 配置。
- 不在代码中写死生产地址或凭据；新增配置同步 `.env.example`。
- 仅在确实读取结果时保留任务结果，避免无意义地占用 Redis。
- 本地关闭第三方服务时，验证码与欢迎邮件流程不得投递 Celery 任务；开发验证码 cache 使用 `LocMemCache`，不连接 Redis。

## 失败与测试

- 记录任务标识、业务对象和异常，不记录密码、令牌等敏感信息。
- 单元测试直接调用任务的 `run` 或底层服务；触发方测试 mock `.delay()`，不连接真实 Redis。
- 需要集成验证时使用独立测试队列，不复用生产 broker。
