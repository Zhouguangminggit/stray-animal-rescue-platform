# Django 开发规范

## 应用与接口

- 一个 app 对应一个清晰业务域；跨域通用代码放入 `apps/core/`，不要建立无边界的工具集合。
- URL 必须设置 `app_name` 并使用命名路由；跳转和模板链接使用 `reverse` 或 `{% url %}`。
- 视图负责 HTTP 编排，复杂业务规则放服务函数；表单负责输入验证，不在模板中实现业务判断。
- 所有修改数据的请求启用 CSRF 保护，受限页面显式增加权限或登录检查。

## 模型

- 字段明确声明长度、默认值、是否允许为空；数据库约束优先于仅在 Python 层校验。
- 为查询条件、外键和唯一业务键建立必要索引，避免无依据的冗余索引。
- 模型变更必须提交 migration、测试和 `db/*.sql`，具体规则见[数据库规范](database.md)。

## 模板与静态资源

- 页面继承 `templates/layouts/application.html`；app 模板使用 `<app_name>/` 命名空间。
- 项目级模板只使用 `layouts/` 和 `components/` 语义命名空间，不创建 `base.html`、`home.html` 等会抢占第三方模板的通用文件。
- 模板只做展示和轻量条件判断，业务数据在视图中准备。
- 基础、共享组件、业务页面样式分层存放；业务 CSS/JS 通过 `extra_css`、`extra_js` 按需加载，不在模板中堆积内联实现。

## 兼容性

保持 Python 3.10+ 与 Django 4.2 兼容。除非需求明确，不改变现有 URL 名称、模板路径或认证行为。
